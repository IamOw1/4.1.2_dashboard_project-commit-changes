# Что ещё не завершено (roadmap) — v4.0

Этот файл помогает понять, какие части идеи уже реализованы в коде, а какие пока остаются задачами “следующего шага”.

## Уже реализовано в текущей версии
- CoreAgent: цикл perceive -> decide -> act, базовая аварийная логика и применение команд
- SubAgent: обработка задач и рекомендации (без замены CoreAgent принятия решений)
- PitControllers: симуляционная модель моторов + ARM/DISARM/TAKEOFF/LAND/HOVER/SET_THROTTLE
- MeshNetwork: UDP обмен сообщениями, discovery/heartbeat (упрощённо)
- OpenQ: запись полётных логов, анализ и экспорт (часть функций зависит от того, запускали ли запись)
- FastAPI REST API + WebSocket `/ws/telemetry`
- Запуск через `main.py` и через `docker-compose`
- Unit-тесты (`pytest`) для ключевых модулей

## Пока не реализовано полностью (что стоит улучшить)
1. Полноценное выполнение миссий через REST API
   - `/api/v1/mission/start` в текущем виде возвращает `mission_id` и количество waypoint’ов, но не запускает реальное выполнение миссии в агенте.
   - Требуется связать endpoint с `CoreAgent.run_mission()` и добавить остановку/статусы миссии.

2. UI миссий (Streamlit dashboard)
   - Экран “Миссии” сейчас подтверждает действия пользователю, но не синхронизирован с backend-исполнением миссий.

3. PyQt GUI как “живой” фронтенд
   - Часть отображений использует внутренние демо-данные.
   - Желательно подключить GUI к API/WS: телеметрия, команды, состояние моторов и OpenQ-статус.

4. Модульная “tool”-система с endpoint’ами
   - В `config/config.yaml` описаны “tools”, но в текущем коде нет полноценной системы загрузки tools и endpoint’ов вида `/api/v1/tools/...`.

5. Аутентификация/авторизация для API
   - Для production-использования стоит добавить защиту: API key/JWT и rate limiting.
   - В текущем коде маршруты без ограничений.

6. Тонкая настройка зависимостей
   - Для людей, которые хотят только API/симуляцию/тесты, можно сделать “легкие” наборы зависимостей (extras), чтобы не тащить тяжёлые ML-компоненты без необходимости.

## Статус дальнейших работ
- Приоритет: пункты (1)-(3), потому что они напрямую влияют на “end-to-end” пользовательский сценарий.
